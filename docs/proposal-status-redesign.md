# 設計提案：マッチングステータスの再定義について

> 作成者：小野寺 香織（キャプテンズ）
> 宛先：吉田 航平 様（よしこさん）
> 作成日：2026-05-02

---

## 提案概要

マッチング申請のステータスを、**事務局スタッフの業務観点に基づいた4状態**に再定義することを提案します。RFP §4-E に記載の3値からの変更となるため、ご確認をお願いします。

---

## 1. 提案内容

### Before（RFP §4-E 記載）

3値で定義：
- **受付**
- **相談会割当済**
- **クローズ**

### After（提案）

業務観点で4値に再定義：

| ステータス | 意味 | 事務局のアクション |
|---|---|---|
| **待機中** | 申込が来たが、まだ4件未満（`受付` の一部） | 様子見（何もしない） |
| **⚠ 要対応** | 4件以上集まった、相談会企画の判断時期（`受付` の一部） | **相談会の調整を開始** |
| **開催準備中** | 相談会の日程決定済み、開催待ち（`相談会割当済` 相当） | リマインド送信など |
| **終了（理由）** | 開催完了 or 企業辞退 or 期限切れ（`クローズ` 相当） | 履歴として記録 |

---

## 2. 提案する理由

### A. 事務局スタッフが直感的に動ける

- **「受付」**：ユーザーから申込が来ただけで、**動くべきかどうかが分からない**
- **「要対応」**：見ただけで「**今すぐ動く案件**」と分かる

事務局スタッフ（5〜10名、ITリテラシー中程度）が**朝のダッシュボード確認 → アクション判断**を5秒で完了できる UX 設計。

### B. 画面表示と内部データの一貫性

仮に内部データが「受付」のままで、画面だけ「要対応」と表示する設計だと：
- エンジニアの認知コストが上がる（「なぜ画面に表示されない値がデータにある？」）
- 通知ロジック等で「受付 + 件数 4以上」の条件分岐が必要
- データと UI で用語が二重管理になる

統一することで、**システム全体で1つの言葉**で会話できます。

### C. ダッシュボードとの一貫性

ダッシュボードの「マッチング申込状況」ドーナツチャートは、すでに4状態（待機中/要対応/開催準備中/終了）で表示しています。マッチング詳細画面の用語と揃えることで、画面間の**用語の不一致**をなくします。

---

## 3. デメリット・懸念点

### A. RFP §4-E との表現差異

RFP では「受付 / 相談会割当済 / クローズ」と明記されているため、**仕様変更の合意**が必要です。

### B. LINE Bot 側のデータとの整合

もし LINE Bot 側のシステムが「受付」というステータス値で動いている場合、**API 連携時に値の変換**が必要になります（マッピングは単純なので開発コストは低）。

### C. しきい値変更時の再分類

「4件 → 5件」のように要対応のしきい値が変わった場合、既存データのステータスを再計算する必要があります（バッチ処理で対応可能）。

---

## 4. 実装上の対応方針（合意の場合）

1. データ層：`status` の値を `待機中 | 要対応 | 開催準備中 | 終了` に
2. 内部の理由詳細：`closed_reason` で「企業辞退 / 期限切れ」等のサブ情報
3. UI 表示：データそのものを表示（UI ⇄ データの変換不要）
4. ヘルパー関数 `getMatchingDisplayLabel()` で「終了（企業辞退）」のような付加表示

すでにプロトタイプには反映済みです。

---

## 5. RFP 通りに戻す場合の対応

もし「RFP §4-E 通りで進めたい」「LINE Bot 側との互換性優先」とのご判断であれば、5/8 までに切り戻し可能です（モックデータと labels の差し替えのみ、所要15分程度）。

---

## 6. ご判断のお願い

下記のいずれかでご回答ください：

- **A. 提案通り（4状態でデータも UI も統一）**
  → このまま 5/3 納品

- **B. RFP §4-E 通り（3状態：受付 / 相談会割当済 / クローズ）に戻す**
  → 5/8 納品で切り戻し対応

- **C. ハイブリッド（データは3値、UI は4状態の表示）**
  → 切り戻し作業の一部のみ実施

ご相談・ご質問あればお気軽にどうぞ。
